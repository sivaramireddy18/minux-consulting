# Weekly Execution Plan: Week 03

## 🎯 Weekly Goal
Master the C compilation pipeline, understand variable properties (scope, lifetime, linkage), utilize storage class specifiers (`static`, `extern`, `volatile`, `const`), and grasp the low-level behavior of compiled symbols in standard object files.

## 🏆 Weekly Outcome
By Friday, the student will be able to dissect a complex multi-file C project, map out exactly how code translates from preprocessing through linking, identify symbol scopes (`static` vs global symbols), and write program blocks that control variable lifetimes across files without leaking scopes.

## 📅 Session Breakdown (Monday - Friday)

### Monday: The C Preprocessor (Translation Units & Macros)
* **Conceptual Objective**: Understand what happens inside the preprocessor. Study header file inclusions (`#include`), conditional compilation (`#ifdef`, `#ifndef`, `#else`, `#endif`), header guards to prevent multiple definition errors, and macro definitions (object-like and function-like macros).
* **Practical Activity**:
  1. Write a C file with multiple nested headers and inspect the preprocessed output file (`.i`) generated by compiling with `-E`.
  2. Implement robust macro guards and note the elimination of duplicate declarations.
* **Code/Circuit Reference**:
```bash
# Compile to preprocess stage only
gcc -E src/main.c -o build/main.i

# Check line directives and header expanded code inside the .i file
cat build/main.i | head -n 50
```

### Tuesday: Compilation & Assembly Phases
* **Conceptual Objective**: Study the compilation pipeline: translation from C to assembly representation (`.s`), optimization parameters (`-O0`, `-O1`, `-O2`, `-O3`, `-Os`), and translation of assembly to machine instructions inside binary object files (`.o`).
* **Practical Activity**:
  1. Compile a simple loop function under different optimization levels (`-O0` vs `-O2`).
  2. Inspect the resulting assembly code (`.s`) and compare how register usage and loop structures are optimized by the compiler.
* **Code/Circuit Reference**:
```bash
# Generate assembly file from C
gcc -S -O2 src/math.c -o build/math.s

# Trace assembly instructions
cat build/math.s
```

### Wednesday: Variables Scopes, Lifetimes, and Linkage
* **Conceptual Objective**: Master variable scopes (block/local, file/global) and lifetimes (automatic/stack, static/data, dynamic/heap). Understand Linkage: internal linkage (accessible only within one translation unit) vs external linkage (accessible across files).
* **Practical Activity**:
  1. Write a program containing global variables, local variables, and static block variables.
  2. Trace variable addresses to observe how they map to different locations in virtual memory.
* **Code/Circuit Reference**:
```c
int global_var = 10;          // External linkage, static lifetime
static int file_var = 20;     // Internal linkage, static lifetime

void function(void) {
    int local_var = 30;       // No linkage, automatic stack lifetime
    static int stat_var = 40; // No linkage, static lifetime
}
```

### Thursday: The Power of keywords - Volatile and Const
* **Conceptual Objective**: Understand the `volatile` qualifier (tells compiler never to optimize away reads/writes because value can change outside program flow, i.e., memory-mapped hardware registers). Study `const` (defines read-only values). Learn combination behaviors: `volatile const` pointers.
* **Practical Activity**:
  1. Write a polling delay loop. Compile with `-O3` without `volatile`. Observe the compiler stripping away the loop in assembly.
  2. Declare the loop counter variable as `volatile` and observe the loop preserved.
* **Code/Circuit Reference**:
```c
// Polling a hardware status register safely
#define UART_STATUS_REG  ((volatile uint32_t*)0x40011000)

void wait_for_data(void) {
    // Without volatile, the compiler loads the register value once
    // and loops infinitely on that cached CPU register!
    while ((*UART_STATUS_REG & 0x01) == 0) {
        // Wait...
    }
}
```

### Friday: Static and Extern in Multi-File Architectures
* **Conceptual Objective**: Grasp the dual role of `static` (restricts symbol visibility to the current file when applied globally; preserves variable value across function calls when applied locally). Understand `extern` (declares existence of symbol defined in another file).
* **Practical Activity**:
  1. Build a C modular layout where internal functions are protected using `static` declarations.
  2. Expose specific API control structures to `main.c` via headers using `extern` correctly.
* **Code/Circuit Reference**:
```c
// sensor.c
static int raw_sensor_reading; // Private global variable (internal linkage)

int get_sensor_value(void) {   // Public function (external linkage)
    return raw_sensor_reading;
}
```

## 🛠️ Hands-On Assignment: Modular State Machine Engine

### Functional Requirements
Create a multi-file C project that implements a state machine utilizing modular files, `static` encapsulation, `extern` interfaces, and strict build optimizations.
1. The project must have three files: `main.c`, `state_machine.c`, and `state_machine.h`.
2. `state_machine.c` must encapsulate the current state using a private file-scope static variable. The variable must *not* be accessible from `main.c`.
3. Provide a public API in `state_machine.h` to update and read the state.
4. Integrate a simulated hardware interrupt tick. The interrupt tick must modify a `volatile` global state change flag.
5. Provide a Makefile that compiles all files with `-Wall -Wextra -Werror -O2`, generates the intermediate preprocessor (`.i`), assembly (`.s`), and object files (`.o`) in a `build/` directory, and links them into a final executable.

### Structural Requirements & Code Skeleton
**`state_machine.h`**:
```c
#ifndef STATE_MACHINE_H
#define STATE_MACHINE_H

typedef enum {
    STATE_IDLE,
    STATE_RUNNING,
    STATE_ERROR
} system_state_t;

// Expose access APIs
void state_init(void);
void state_update(void);
system_state_t state_get(void);

// Expose hardware flag via extern (safe usage declaration)
extern volatile int g_hw_interrupt_flag;

#endif // STATE_MACHINE_H
```

**`state_machine.c` (C-Skeleton)**:
```c
#include "state_machine.h"

// Private file-scoped variables
static system_state_t current_state = STATE_IDLE;
static int transition_counter = 0; // Tracks transitions privately

// Global volatile flag defined here
volatile int g_hw_interrupt_flag = 0;

void state_init(void) {
    current_state = STATE_IDLE;
    transition_counter = 0;
}

void state_update(void) {
    if (g_hw_interrupt_flag) {
        g_hw_interrupt_flag = 0; // Clear flag
        
        switch (current_state) {
            case STATE_IDLE:
                current_state = STATE_RUNNING;
                transition_counter++;
                break;
            case STATE_RUNNING:
                current_state = STATE_IDLE;
                transition_counter++;
                break;
            case STATE_ERROR:
                // Require external system reset
                break;
        }
    }
}

system_state_t state_get(void) {
    return current_state;
}
```

## 💡 Motivation Checkpoint
A major difference between standard application C coding and high-performance embedded systems programming is the precise control over where data is stored in memory and how the compiler behaves. In microcontrollers, RAM size can be as small as 8KB. Knowing how to restrict symbol visibilities, limit global clutter, and secure register actions via `volatile` is the baseline expectation of systems developers.
