## 🎯 Weekly Goal

* **Conceptual Objective**: Master the core physics of pointers. Understand pointer scales (adding $1$ to an integer pointer advances address by 4 bytes). Learn double-pointers (`char **argv`), dereferencing pipelines, and void pointers (`void*`) for generic data routing.
* **Practical Activity**:
  1. Write a program using pointer arithmetic to manually parse through multidimensional arrays and structures.
  2. Implement dereferences to print data layouts byte-by-byte.
* **Code/Circuit Reference**:
```c
int array[5] = {10, 20, 30, 40, 50};
int *ptr = array;

// Pointer arithmetic advancements
printf("Address of index 0: %p\n", (void*)ptr);
printf("Address of index 1: %p\n", (void*)(ptr + 1)); // Advance by sizeof(int) = 4 bytes
printf("Value at index 2: %d\n", *(ptr + 2));       // Equivalent to array[2]
```

### Tuesday: The C Application Memory Map
* **Conceptual Objective**: Study the memory model of an executing binary:
  - **Text Segment**: Machine code (read-only).
  - **Data Segment**: Initialized global and static variables.
  - **BSS Segment**: Uninitialized global and static variables (cleared to 0 at boot).
  - **Stack**: Local variables, return addresses, and stack frames.
  - **Heap**: Dynamic memory.
* **Practical Activity**:
  1. Write a program with variables located in every segment.
  2. Print the memory address of each variable and draw a corresponding memory map diagram showing stack and heap directions.
* **Code/Circuit Reference**:
```text
C Virtual Memory Layout:
  +---------------------------+  High Addresses (0xFFFFFFFF)
  | Stack (Grows Downward)    |
  |             |             |
  |             v             |
  |     [Free Space]          |
  |             ^             |
  |             |             |
  | Heap (Grows Upward)       |
  +---------------------------+
  | BSS Segment (uninit vars) |
  +---------------------------+
  | Data Segment (init vars)  |
  +---------------------------+
  | Text Segment (Read-only)  |
  +---------------------------+  Low Addresses (0x00000000)
```

## 🛠️ Hands-On Assignment: Raw Static Array Memory Allocator

### Functional Requirements
Create a custom, high-reliability memory allocation library (`my_alloc.c`, `my_alloc.h`) that partitions and manages a static memory pool array of 4KB ($4096$ bytes) using a raw block linked list.
1. The library must implement two core functions: `void* my_malloc(size_t size)` and `void my_free(void *ptr)`.
2. Do *not* use standard library `<stdlib.h>` functions (`malloc`, `free`, `realloc`).
3. Every allocated block must be preceded by a `block_header_t` block metadata node.
4. `my_malloc` must use a **First-Fit** algorithm: search the linked block list for the first free block of sufficient size. If the block is significantly larger than requested, split the block into an allocated chunk and a remaining free chunk.
5. `my_free` must mark the block as free. Additionally, it must scan adjacent blocks and **coalesce (merge)** consecutive free blocks to prevent fragmentation.
6. The allocator must enforce memory alignment (align all allocated addresses to 4-byte boundaries).

### Structural Requirements & Code Skeleton
**`my_alloc.h`**:
```c
#ifndef MY_ALLOC_H
#define MY_ALLOC_H

#include <stddef.h>
#include <stdbool.h>

void my_alloc_init(void);
void* my_malloc(size_t size);
void my_free(void *ptr);
void my_alloc_dump(void); // Utility to print layout of heap memory

#endif // MY_ALLOC_H
```

**`my_alloc.c` (C-Skeleton)**:
```c
#include "my_alloc.h"
#include <stdio.h>

#define POOL_SIZE 4096
#define ALIGNMENT 4
#define ALIGN(size) (((size) + (ALIGNMENT - 1)) & ~(ALIGNMENT - 1))

typedef struct block_header {
    size_t size;
    bool is_free;
    struct block_header *next;
} block_header_t;

#define HEADER_SIZE sizeof(block_header_t)

// Memory pool array
static uint8_t memory_pool[POOL_SIZE];
static block_header_t *free_list_head = NULL;

void my_alloc_init(void) {
    free_list_head = (block_header_t*)memory_pool;
    free_list_head->size = POOL_SIZE - HEADER_SIZE;
    free_list_head->is_free = true;
    free_list_head->next = NULL;
}

void* my_malloc(size_t size) {
    size_t aligned_size = ALIGN(size);
    block_header_t *current = free_list_head;

    while (current != NULL) {
        if (current->is_free && current->size >= aligned_size) {
            // Can we split the block?
            if (current->size >= aligned_size + HEADER_SIZE + ALIGNMENT) {
                block_header_t *new_block = (block_header_t*)((uint8_t*)current + HEADER_SIZE + aligned_size);
                new_block->size = current->size - aligned_size - HEADER_SIZE;
                new_block->is_free = true;
                new_block->next = current->next;

                current->size = aligned_size;
                current->next = new_block;
            }
            current->is_free = false;
            return (void*)((uint8_t*)current + HEADER_SIZE);
        }
        current = current->next;
    }
    return NULL; // Out of memory
}

void my_free(void *ptr) {
    if (ptr == NULL) return;

    block_header_t *header = (block_header_t*)((uint8_t*)ptr - HEADER_SIZE);
    header->is_free = true;

    // Coalesce free blocks
    block_header_t *current = free_list_head;
    while (current != NULL && current->next != NULL) {
        if (current->is_free && current->next->is_free) {
            current->size += HEADER_SIZE + current->next->size;
            current->next = current->next->next;
        } else {
            current = current->next;
        }
    }
}
```

## 💡 Motivation Checkpoint
In high-reliability embedded system architectures (like automotive ISO-26262 systems), calling the standard C library `malloc()` or `free()` is banned. This is due to the non-deterministic execution times of typical allocators and the risk of memory fragmentation locking up the system. Knowing how to construct dynamic, deterministic block allocators or pool managers is a core requirement for mission-critical firmware developers.

## 📝 Trainer Review Rule
* **Pass Criteria**:
  - Allocator splits blocks correctly (dump shows residual free blocks created on allocation).
  - Freeing two consecutive blocks successfully merges them into a single larger block.
  - Aligned data offsets are strictly divisible by 4.
  - Zero compiler warnings or warnings of memory leak inside test application.
* **Fail Criteria**:
  - Direct call to standard `<stdlib.h>` malloc inside implementation.
  - Misaligned memory offsets.
