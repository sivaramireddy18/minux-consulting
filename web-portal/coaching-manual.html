Vetting Directive

Operating Rules & Evaluation Standards

Establishing absolute rigor. We do not evaluate using loose curves: our candidates are graded against high-stakes industrial safety benchmarks.

The Candidate-Client Interface Laws

Rule 01

The Zero-Warning Directive

Every line of C code written during this vetting process must compile under GCC/Clang cleanly using -std=c11 -Wall -Wextra -Werror -pedantic. If a single warning is emitted, compilation aborts, and the submission is rejected immediately without further audit.

Rule 02

Zero-Leak Dynamic Memory Policy

Dynamic memory allocation must be managed with absolute precision. All compiled binaries are executed through Valgrind Memcheck. If a single byte of memory is leaked, or if an uninitialized memory read or invalid free is detected, the candidate fails the milestone evaluation automatically.

Rule 03

Defensive Coding & Standards (MISRA C Subset)

Safety-critical code requires strict standards. Standard library calls that are susceptible to stack buffer overflows (e.g. strcpy, sprintf, gets) are strictly banned. Candidates must use bounded alternatives (strncpy, snprintf) and adhere to MISRA C safety-critical principles.

Rule 04

Strict Python Type Quality

In python automation scripts, type omissions are forbidden. All code must incorporate explicit static type annotations, validated under strict Mypy checks. Connection handles and serial sockets must implement context managers to prevent open-resource locks.

The 10-Point Technical Score Rubric

Rating Band Technical Criteria Outcome Status
9 to 10 Points Perfect algorithmic correctness, optimized memory margins, 100% test coverage, and complete execution of the highly complex Stretch Task. Pass with Distinction
