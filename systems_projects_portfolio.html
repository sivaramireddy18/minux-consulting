🏆 Project 1 (C Systems): MinuxDB - Concurrent Key-Value Database Engine

An elite, high-performance, disk-persisted concurrent Key-Value Database Engine written in pure C11, bypassing standard high-level SQL database libraries.

🏛️ System Architecture

graph TD
    Client["Client Thread (REPL/API)"] -->|Submit Command| Parser["Command Parser & Validator"]
    Parser -->|Aligned Allocation| Arena["Memory Arena / Pool Manager"]
    Parser -->|Write-Ahead Log| Disk["Unbuffered I/O File Descriptor (.log)"]
    Parser -->|Mutex Protected Write| Index["In-Memory Index Tree (Hash/B-Tree)"]
    Index -->|Concurrency Control| RWLock["POSIX Read-Write Locks"]

🛠️ Core Tech Stack & Requirements

  • Language/Standard: Strict ISO C11 (-std=c11 -Wall -Wextra -Werror -pedantic).
  • Memory Management: Custom 8-byte aligned Memory Arena (for transient parse tokens) and Object Pool (for database index nodes). Standard malloc strictly banned in execution loops.
  • Systems I/O: Direct, unbuffered file descriptors (open, read, write, close, fsync) for Write-Ahead Logging (WAL).
  • Concurrency: POSIX Threads (pthreads), Read-Write Locks (pthread_rwlock_t) for thread-safe concurrent index lookups.
  • Safety: 100% MISRA C compliance subset, validated via cppcheck. Zero memory leaks under Valgrind.

🏃‍♂️ Sprint 1: Memory Architecture & Basic Storage

  • Deliverable: arena.h/arena.c containing arena_alloc with compile-time assertions.
  • [JIRA-CDB-102] Object Pool Node Allocator
  • Description: Implement a block allocator using free-list bitmaps to serve database entry nodes in constant O(1) time.
  • Deliverable: pool.h/pool.c with 0 external fragmentation.
  • [JIRA-CDB-103] Unbuffered Write-Ahead Log (WAL) Engine
  • Description: Write a persistent logger that records database insert operations directly to disk via raw system descriptors, forcing disk sync (fsync).
  • Deliverable: wal.c managing crash-resilient append-only logs.

🏃‍♂️ Sprint 2: Database Indexing & Command Parsing

  • [JIRA-CDB-201] High-Speed In-Memory Hash Index
  • Description: Build an in-memory hash table using separate chaining for collision resolution, storing key-to-offset maps.
  • Deliverable: index_hash.c supporting O(1) key lookups.
  • [JIRA-CDB-202] Branch-Free Command Tokenizer
  • Description: Write a state-machine based command string parser that extracts commands (GET, PUT, DEL) with cyclomatic complexity <= 3.
  • Deliverable: parser.c parsing string queries into structured execution models.

🏃‍♂️ Sprint 3: Concurrency & Lock Disciplines

  • [JIRA-CDB-301] Dynamic Read-Write Locks Integration
  • Description: Protect the Hash Index using POSIX RW Locks, enabling concurrent read lookups but mutually exclusive write modifications.

  • [JIRA-CDB-302] Active Transactions Crash Recovery Manager
  • Description: Implement a recovery boot routine that scans the WAL file, parses uncommitted transactions, and rebuilds the hash index automatically.
  • Deliverable: recovery.c verifying index reconstruction on dirty boots.

🏃‍♂️ Sprint 4: Performance Audits & Verification

🏆 Project 2 (Embedded): MinuxGateway - Real-Time Industrial IoT Gateway

A real-time, closed-loop telemetry gateway utilizing FreeRTOS on an STM32 microcontroller to capture analog sensors data, communicating via DMA-UART with a custom Embedded Linux character device module running on a Raspberry Pi.

🏛️ System Architecture

+--------------------------------------+      UART DMA      +------------------------------------------+
|       STM32 MCU (FreeRTOS)           |     Telemetry      |        Raspberry Pi (Embedded Linux)     |
|  [ADC Task] -> [Queue] -> [TX Task]  |------------------->|  [Kernel Module Driver] -> [/dev/minux]  |
|         ^ (Hardware Interrupt)       |      Channel       |                 |                        |
|  [Emergency Button IRQ]              |                    |  [User Space Daemon] -> [Data Logging]   |
+--------------------------------------+                    +------------------------------------------+

🛠️ Core Tech Stack & Requirements

  • MCU Firmware: FreeRTOS ports, direct register manipulation RCC/GPIO/USART, NVIC nested interrupts, DMA channel control, STM32F4.
  • Kernel Space: Out-of-tree Linux Kernel Module (LKM), character device registration (cdev), file operations (file_operations), user-kernel space copy (copy_to_user), Device Tree compatible string mapping.
  • Hardware Interfacing: Direct UART TTL wired bridges, logic analyzers for frame debugging.

🏆 Project 3 (Automation): MinuxHIL - Enterprise HIL Automation Framework

A fully typed, modular, and concurrent Hardware-in-the-Loop (HIL) Test Automation Framework written in pytest, incorporating PyVISA SCPI drivers factories, relay fault injections, binary ELF parsing, and automated Jenkins pipelines.

🏛️ System Architecture

graph TD
    TestRunner["pytest Engine (conftest.py)"] -->|Dynamic Selection| DriverFactory["Device Driver Factory"]
    DriverFactory -->|VISA/SCPI connection| PowerSupply["Singleton Power Supply Wrapper"]
    DriverFactory -->|Serial USB interface| RelayBoard["Relay Controller (Fault Injection)"]
    TestRunner -->|Queue Telemetry| LogThread["Asynchronous Logging Thread"]
    TestRunner -->|Post-Test Compile| PDFReport["Weasyprint HTML-to-PDF Engine"]
    TestRunner -->|Archiving| CI["Jenkins / GitHub Actions pipeline"]

🛠️ Core Tech Stack & Requirements

  • Framework: pytest, parameterized testing matrices, conftest.py fixtures injection.
  • Interfacing: PyVISA SCPI protocols, USB Relay serial, PySerial byte parsing.
  • Type Safety: 100% strict type hints (mypy --strict compliance).
  • Firmware Audits: ELF section sizes and symbol table analysis (pyelftools).
  • CI/CD: Jenkinsfile / GitHub Actions pipeline, JUnit XML parsed results summaries, Slack notification webhooks.
